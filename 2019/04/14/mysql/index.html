优化设计

总结一下，举例：

设计一张表。该表中存在这些字段：

  • user_id: 评论用户ID
  • content: 评论用户内容
  • time: 评论的时间 （这个可以根据评论的时间排序，额也可以按照点赞数量排序。再说吧） 2017-08-07 14:45:30
  • replay_id: 回复（也就是被评论）用户的ID （如果回复作者第一层级的话，可以设计为空。 额，其实也可以赋值为作者ID，这样比较耗内存，最好不要这样写吧）
  • depth: 评论内容的层级。 这是最近看到的，这个有俩点好处。其一，展示的时候用户一般不需要看全部回复内容，我们可以控制显示数量。比如我只显示回复层级不超过五个深度层级。用户如果感兴趣，可以继续点击查看更加深度回复内容。这个类似于segmentfault社区，形成一个树形结构。其实这也需要前端展示需求。因为如果页面仅仅展示俩层关系的话，类似B站，因为我发现B站是根据用户点赞数量进行排序的，如果点赞数量相同的话则根据时间排序，所以不存在什么深度问题。当然，这里我使用depth展示，问就是三个字，简单！！！ 追加： 这里会出现多层级问题，前端页面展示奔溃问题。目前我还是修改了，不需要depth。刚与同学交流后，学会了更加方便的一种方式。采用绑定内容唯一ID的方式。每一行评论内容都绑定唯一ID，这个ID就是下一级评论指向的ID，这样就可以递归嵌套获取层级，从而取消了depth关系，而且这样更好，因为depth无法判断父级ID
  • id: 内容ID， 每一行制定的唯一ID，主键
  • type: 判断是否是评论文章的。 这个字段是递归查询的初始入口数据。
  • praise_num: 点赞数量 根据这个排序 点赞 》时间
  • is_valid： 追加字段， 判断是否有效， 这是用户删除评论使用。如果删除，则不展示，但是也不删除。递归非常耗时间，还不如直接保留，毕竟评论也不会占据太多容量的。 后期可以尝试类似QQ微信的做法，直接删除前六个月的所有记录。根据时间删除，不需要递归操作。
