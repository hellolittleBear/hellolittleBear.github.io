community-需求设计

该项目最初是为毕业设计答辩的项目，后来为了设计一个专门的开发者社区，与该项目功能类似，所以在该项目的基础上优化改版而来。也是前端的敲门砖吧~
希望这块砖一直在“加量不加价”。

核心功能

解决github、掘金、SF的痛点就是核心功能。

网站分析亮点和痛点

掘金

  • 内容品类残缺不全，部分品类展示权重过高，而偏门或者高深的品类文章权重偏低

在多种品类的文章差距过大，热门和简单的品类数据充斥着大量的文章，对于其他冷门品类不友好。应当适当设置天花板降低热门文章数量和提高冷门文章福利。

  • 推送精准度不高，用户体验感降低，普遍推送文章深度较低，而且类型大多相同

推送初始进入该社区应该针对用户提供多种选择，时间推送还是根据用户擅长技能推送，且推送文章更新频率适当改变。
可根据用户等级推送，相关等级的文章。

  • PC端UI设计过于简单化

UI综合考虑SF、掘金、github

  • 广告太多，严重影响用户体验，尤其对于中高级开发者

广告区域只存在某些特定区域，并且专门可以设置一个广告区域。用户可以设置广告的发布类型和布局。也可在后台根据用户的擅长技能适当发布相关类型的广告。

  • 用户与社区管理交流建议窗口体验较差，未提供有效窗口共建社区

专门提供一个issue窗口，并且根据用户提出的issue有效的奖励机制。

  • 评论体验感较差

评论采用哔哩哔哩移动端的评论展示方式，并且增加特设的emoj，参考知乎

  • 用户主页信息量非常少，对用户的信息整理不够全面

用户主页参考github和SF

  • 文章标题普遍存在UC风格，评审质量堪忧

可以设置用户发布文章进行分等级，等级一出，并设置一些特定规则，惩罚机制，奖励积分

SF

  • 主打问答，对标stack overflow，却侧重于文章，首页展示文章列表而非问答页面。对标方向偏差

本项目主打文章、问答、个人主页。

  • 品类过于简单化，文章质量较低，高端文章多数转载或者翻译，原创比例较低

动态设置用户翻译、转载文章数量。

  • UI主题视线柔和，用户体验良好

提供多种用户主题风格，亦可以用户提供风格建议或者提供一个主题demo展示，用户自创风格亦可。

  • 首页展示文章按照时间排序，高质量文章泯然于世

首页文章展示由用户控制，时间展示、质量展示、技能展示等等

  • 存在低端文章泛滥，中高品质文章稀缺

参照掘金处理方式。

  • 文章标题普遍存在UC风格，评审质量堪忧

参考掘金处理方式。

  • 个人主页根据声望排名，声望通过问答和讲堂进行计算

扩展计算方式，并增加惩罚机制

  • 专栏分类的文章推荐、热门、最新三大模块基本不可信，含金量靠钱和运气堆，虽然钱的“含金量”很高~

专栏主要根据高级用户的发布文章，低中级用户发布文章希望渺茫。设置用户每周自荐文章，由多个大佬判决改文章是否能够入推荐榜单。

  • 圈子全权管理员控制，权利分配出现问题，导致发布圈子信息缓慢

暂时关闭圈子，等待其余工作完成再行分析需求和设计。

  • 显示机制存在较大漏洞，文章展示一直是通过点击数量排序的

文章热门通过评分和点击量进行排名

github

  • 提交contribute展示亮点，但是存在用户故意刷issue而一片绿的效果，检测每次提交是否合理有效

设置有效的机制进行计算，例如评论被点赞、问答被参考有效、文章被推荐和点赞都直接影响contribute，而不是单纯的git commit、push而已。

  • star存在恶意刷，提高star、fork、watching的权限

个人主要提供用户展示自己的开源项目或者其他作品

others issues

  • 开发者初始进入社区必须注册且必须使用电话号码和邮箱
  • 每个用户每周有一次机会自荐一篇文章在专栏上。有管理团队进行筛选
  • 管理团队由各个用户中高级开发者组队而成
  • 提供一个公开的平台进行交流。类似于v2ex帖子
  • 为开发者提供多种开发交流环境，EN和CH
  • 专门为文章和问答提供俩种模块，互相分离，提供一种糅合机制，在个人主页展示文章评论和问答评论，类似于SF的声望，就叫AP吧，在AP上设置一个等级，类似于玄幻小说的等级制度，并根据每个等级制度设置一定的奖励机制

细节

  • 编辑文章。

    • 为了更好的提高开发者的编辑能力和文章整理能力，目前仅仅支持markdown编辑模式。
    • 为何出现用户编辑文章过程处于断网或者其他中断编辑状态，需要定时保存文章草稿。
    • 文章存在不同分类、翻译、原创、转载。
    • 实时响应文章编辑后的展示,具体参考cmd markdown。
    • 专刊模块暂时不考虑，目测没有什么实际作用，仅仅是展示所有文章而已。其实在个人主页就可以展示。

  • 文章详情模块

    • 在文章详情模块中，用户可以浏览、点赞、评论回复、举报、分享、赞赏等操作。
    • 针对每一个功能进行细致的划分，并对每一个功能之间相互的关联进行设计归纳。详情如下：

      浏览功能： 借鉴cmd markdown的浏览功能，并且目录与文章动态绑定。
      评论回复功能： 借鉴哔哩哔哩评论设计风格。
      赞赏功能：暂时丢弃，无法完成….

    • 文章列表展示模块： 主要用于首页展示，使用segmentfault设计。

然后在问答模块，具体功能如下：

  • 提问模块

    • 根据问题的来源（开发实战类型、编程题目类型、工具软件类型）、技术类型（开发语言或者其他专业类型语言）进行分类，这些分类在后期的扩展提供更多的功能。用户可以根据这些分类进行筛选。

  • 个人主页模块

    • 主要展示用户基本信息和用户对社区的关联信息。这些信息非常重要，是一个数据综合体。 摈弃segmentfault的RP等级制度，而是借鉴哔哩哔哩等级制度，并进行扩展优化。详情点击

用户等级制度的详解

1、初始用户登录注册时等级最低为0，该用户无法评论、无法回复信息。在文章和提问次数达到一定的数量后，并且通过个人擅长技术考试后才可进行评论。
2、根据用户每日登陆活跃度、笔记记录、文章提问评论的点赞、收藏、举报等等对用户进行等级制度的动态改变。用户等级无法手动修改.
